= ZIP Extractor Decision Record
:toc:

== Define Context

Quarkus Forge currently extracts generated project archives through custom components:

* `SafeZipExtractor`
* `ZipCentralDirectoryReader`
* `ArchiveSafetyPolicy`
* `ProjectArchiveService`

Section 11.1 requires an explicit keep-vs-migrate decision for archive extraction before any migration work.

== Define Options

[cols="1,3",options="header"]
|===
| Option | Summary

| Keep custom extractor
| Continue using current extraction pipeline and hardening controls, with maintenance in-repo.

| Migrate to Apache Commons Compress
| Replace all or part of extraction logic with Commons Compress ZIP APIs.

| Migrate to zip4j
| Replace extraction logic with zip4j ZIP APIs and safety checks layered on top.
|===

== Record Decision

Decision: keep the custom extractor for now.

Rationale:

* Current implementation already enforces required safety guarantees in first-party code:
  ** path traversal and absolute path rejection
  ** symlink and suspicious unix mode rejection
  ** central-directory and payload consistency checks
  ** duplicate normalized-entry rejection
  ** zip-bomb controls (entry count, total extracted bytes, compression ratio)
  ** staging + atomic move/fallback overwrite behavior
* Existing automated coverage is broad and targeted for malformed and adversarial archives.
* Migrating to a third-party extractor would still require re-implementing and re-proving several project-specific safety invariants, with no immediate user-facing benefit for this phase.

== Define Revisit Triggers

Re-open this decision only if one of these occurs:

. A required archive format capability is missing from current implementation.
. Security audit identifies a meaningful weakness that cannot be reasonably fixed in current code.
. Measured maintenance cost of custom extractor exceeds agreed project budget.
. A migration proposal includes parity tests proving all current guarantees remain intact.
