= Cross-Platform Validation Matrix
:toc:

== Purpose
Define platform profiles, terminal backend expectations, deterministic validation scenarios, and known fallback constraints.

== Backend Policy
Current runtime backend selection policy:

* With native access flag `--enable-native-access=ALL-UNNAMED`: prefer `panama,jline3`.
* Without native access flag: prefer `jline3` and emit guidance about native-access setup.
* `tamboui.backend` / `TAMBOUI_BACKEND` may override backend preference when explicitly set.

== Supported Matrix
[cols="1,1,2,2",options="header"]
|===
| Profile | OS family | Expected backend preference | Required validation focus

| P1
| Linux (glibc)
| `panama,jline3` with native-access flag
| Startup, TUI navigation, generation flow, backend fallback when flag is absent.

| P2
| macOS
| `panama,jline3` with native-access flag
| Startup + keyboard routing parity + generation path + terminal rendering parity.

| P3
| Windows
| `jline3` expected default in many environments
| CLI/headless behavior parity + path/output validation + JLine fallback stability.
|===

== Deterministic Validation Scenarios
Run these scenarios for each profile:

. `quarkus-forge --dry-run` validates defaults and exits with usage success path.
. Root command starts TUI, focus traversal works (`Tab`/`Shift+Tab`), then exits cleanly.
. Headless generation (`quarkus-forge generate ...`) preserves exit-code mapping (0/2/3/4/130).
. Catalog load fallback remains `live -> cache -> snapshot` under simulated API failure.
. Backend preference reflects native-access flag policy (Panama-preferred vs JLine fallback).

== Known Constraints

* Native-access flag is required for Panama-first preference on JVM runs.
* Without native access, runtime prefers JLine and may still display terminal-native warnings.
* Platform/terminal capability differences can alter rendering quality; behavior semantics must remain stable.
* This matrix defines behavior expectations, not distribution-level support guarantees for every terminal emulator.
